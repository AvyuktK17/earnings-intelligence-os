# Semiconductor Equity-Research Report (analyst-draft skill)

You are drafting a **Claude-assisted draft for analyst review** — a
professional semiconductor earnings-update narrative — using **only** a
deterministic *report packet* as your source of facts.

A report packet is generated by the project's exporter:

```bash
python export_report_packet.py --ticker AVGO --accession-number 0001730168-26-000051
```

It writes a Markdown packet (and a JSON sibling) under
`output/report_packets/`. The packet contains the company profile, the selected
filing's metadata, trusted human-reviewed claims with their source references /
chunk ids / supporting excerpts, reported financial metrics (latest + history),
a peer-comparison table, a dated valuation snapshot, and metadata for any
existing deterministic report.

## Step 0 — Load the packet (and only the packet)

1. Ask the user for the **path to the report-packet markdown file** if they
   have not already provided one (e.g.
   `output/report_packets/avgo_0001730168_26_000051_packet.md`). The JSON
   sibling may be read too if present, but the Markdown packet is sufficient.
2. Read that file. **Every fact in your draft must come from this packet.** Do
   not pull in figures from memory, the web, prior conversations, other
   companies, or any other file. If the packet does not contain a number, you
   do not have it.
3. Read this skill's two supporting files before drafting:
   - `template.md` — the exact section order and what each section must contain.
   - `quality-checklist.md` — the pass/fail checks to self-review against.

   (Load these only now, when the skill is invoked, so routine Claude Code
   sessions stay token-efficient.)

## What to produce

A single Markdown document following `template.md`, with these sections in this
order:

1. Front-page executive summary
2. Quarterly results analysis
3. Guidance and outlook
4. Historical operating trends
5. Peer comparison
6. Balance sheet and free-cash-flow discussion
7. Valuation snapshot
8. Catalysts
9. Risks and watch items
10. Evidence appendix
11. Methodology and limitations

The reference structure (an analyst earnings note: thesis-led summary, then
results / outlook / trends / peers / balance sheet / valuation, then catalysts,
risks, and an evidence trail) is modelled on a conventional sell-side note. Use
that **structure and professional tone only** — do **not** copy any branding,
wording, logos, or the forecasting/DCF/target-price methodology that such notes
typically contain. Those are prohibited here (see guardrails).

## Mandatory guardrails (non-negotiable)

- **Never invent a rating** (no Buy / Hold / Sell / Overweight / etc.).
- **Never invent a target price** or implied/fair-value share price.
- **Never fabricate forecasts** or forward estimates of any metric.
- **Never generate DCF outputs** (no WACC, terminal value, implied value).
- **Never invent consensus estimates** or "vs. Street" comparisons.
- **Clearly label valuation data as a dated, manually reviewed snapshot** — not
  a live market feed — every time you cite it.
- **Distinguish reported facts from analyst interpretation.** State reported
  figures plainly; clearly mark any synthesis, framing, or judgement as
  interpretation (e.g. "Interpretation:" or hedged language). Never present
  interpretation as if it were a reported number.
- **Cite the supplied source references and chunk ids** for every claim you draw
  from the packet's trusted claims (e.g. "(source: <source_reference>, chunk
  <id>)").
- **Preserve missing values honestly.** If the packet labels something
  "Not available", say it is not available — do not estimate, interpolate, or
  fill it in.
- **Label the document** at the very top with the exact line:
  `Claude-assisted draft for analyst review`.
- **Do not write into trusted database tables**, call any API, upload the draft,
  or run any command that mutates state. This skill produces a **local draft
  only**. Saving the draft as a local `.md` file (e.g. next to the packet) is
  fine; persistence into Supabase or any service is out of scope.

## How to handle thin or empty sections

- If there are no trusted claims for catalysts or risks, say so plainly rather
  than inventing items — e.g. "No trusted, human-reviewed claims in this packet
  support catalysts." Optionally note neutral, fact-based watch items derived
  strictly from reported figures, clearly marked as interpretation.
- If a whole data area is missing (e.g. no valuation snapshot), state that the
  packet did not include it and continue.

## Finishing

After drafting, self-review against `quality-checklist.md`. Fix any failures
before presenting the draft. Then present the draft to the user and remind them
it is a **Claude-assisted draft for analyst review** that has not been
human-reviewed or published.
